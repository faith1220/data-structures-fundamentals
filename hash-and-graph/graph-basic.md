# 图的基本概念与存储

在前面的学习中，我们从“一条线”的**线性结构**（数组、链表），学到了“一对多”的**树形结构**（二叉树、堆）。 但是，现实世界的关系往往比“老板和无数个小弟”的树结构还要更加复杂。

想象一下：

1. **微信里的人际关系**：小明和小红互相是好友，小红又是小刚的好友，但小明却不认识小刚。
2. **城市的交通网络**：北京可以飞往上海，上海可以飞往广州，北京也可以直接飞往广州。
3. **互联网的网页链接**：网页A有个链接指向网页B，网页B也有个链接指向网页A。

在这种极其错综复杂&#x7684;**“多对多”**&#x7F51;络关系面前，树形结构也无能为力了。于是，计算机科学中最庞大、最复杂的终极 Boss 登场了 —— **图（Graph）**。

## 图的核心组件：“点”与“线”

不要被“图”这个名字吓到。在数据结构里，一幅图只由两样极其简单的东西组成：

1. **顶点（Vertex / Node）**：表示现实中的实体。比如城市、网页、微信用户。
2. **边（Edge）**：连接两个顶点的线，表示实体之间的某种关系。比如航班路线、超链接、好友关系。

```
 (北京)-------(上海)
    \         /
     \       /
      (广州)
```

## 图的两大门派：有向图 vs 无向图

根据这些“边”是否带有箭头（是否具有单向/双向属性），图被分为两类：

### 1. 无向图 (Undirected Graph)

**特点：边没有箭头，关系是双向的。**

* **生活例子**：微信的“好友关系”。我加你为好友，你列表里必然也有我，这是双向平等的。
* 如果顶点 A 和 顶点 B 之间有一条边，你既可以从 A 走到 B，也可以从 B 退回 A。

### 2. 有向图 (Directed Graph)

**特点：边带有单向箭头，关系是单向的。**

* **生活例子**：微博/Twitter 的“关注关系”或者 B站的“粉丝”。我关注了你（我指向你），但不代表你也关注了我（你不一定指向我）。单行道也是典型的有向图。
* 或者像网页链接：网页A里有个超链接可以点击跳到网页B，但你不一定能从网页B点特定的链接跳回A。

***

## 给线加上代价：带权图 (Weighted Graph)

现实中，从北京飞上海，和从北京飞纽约，虽然两者都有一条“边（航线）”相连，但它们消耗的“代价（时间、机票钱）”是截然不同的。

如果在图的每一条**边**上，都写着一个代表代价的数字（叫做**权重 Weight**），这幅图就升级成了**带权图**。 导航软件里的“查找最短时间路线”，本质上就是在带权图中寻找一条权重总和最小的路线。

## 图是怎么装进内存里的？（图的存储）

这才是初学者觉得最头疼的地方。 数组就是一排格子，二叉树就是连着左右两个小弟，它们在内存里的长相很直觉。 但是，图这种乱七八糟、毫无章法的网状结构，该用什么代码或者变量把它存进硬盘和内存里呢？

前辈们发明了两种极其经典的存储方式：**邻接矩阵** 和 **邻接表**。

### 方式一：邻接矩阵 (Adjacency Matrix) —— 财大气粗的二维数组法

假设我们有 4 个顶点，标号为 0, 1, 2, 3。 我们可以直接开辟一个 `4 x 4 = 16` 个格子的**二维数组（矩阵）**。

* `Matrix[i][j] = 1`：代表顶点 i 到顶点 j 之间**有一条路/边**。
* `Matrix[i][j] = 0`：代表走不通。

假设只有 0 连着 1，0 连着 2，1 连着 3。矩阵长这样：

```
      0  1  2  3  (去往)
   +------------
(0)|  0  1  1  0  -> 0能到1和2，不能到3
(1)|  1  0  0  1  -> 1能回0，能到3
(2)|  1  0  0  0  -> 2能回0
(3)|  0  1  0  0  -> 3能回1
```

_(注意：无向图的矩阵一定是沿着主对角线完全对称的！)_

**邻接矩阵的优缺点：**

* **大优点**：查询极快！想知道点 A 和点 B 之间有没有路？只要一行代码 `if (matrix[A][B] == 1)`，速度为 O(1)！
* **致命缺点**：极度浪费内存！想象一下微信有 10 亿用户，但每个人平均只有 200 个好友。如果你开一个 `10亿 x 10亿` 的二维数组，里面 99.999% 的格子全都是 `0`（没有任何关系的陌生人）。不仅你的电脑内存会瞬间爆炸，这也是极大的算力浪费。

### 方式二：邻接表 (Adjacency List) —— 精打细算的朋友圈列表

由于现实世界中绝大多数图都是“稀疏图”（即一个人不可能认识全世界所有人，一条边只连着极少数几个人）。因此，工业界绝大部分情况下采用的是**邻接表**。

它的思路非常像你的**微信通讯录**：我们只记录每个人**实实在在认识的那些朋友**。

* 顶点 0 的好友列表：`[1, 2]`
* 顶点 1 的好友列表：`[0, 3]`
* 顶点 2 的好友列表：`[0]`
* 顶点 3 的好友列表：`[1]`

在代码层面，这通常体现为\*\*“一个装满了动态数组（或链表）的数组”**，或者是**“一个装满了动态数组的哈希表”\*\*。我们来看四种语言的经典实现：

```python
# Python 中最通常的做法：使用字典（哈希表）来映射每一个顶点的邻接列表
# Key 是顶点，Value 是它的“朋友圈”（动态列表 List）

graph = {

    2: [0],

# 增加一条无向边 (比如让 2 和 3 也认识)

graph[3].append(2) # 因为是无向图，所以要互相添加
```

{% tab title="Java" %}
```java
import java.util.*;

public class Main {
    public static void main(String[] args) {
        // Java 中推荐用法：数组里面装 List，或者 HashMap 里面装 List
        // 这里演示用 List 嵌套 List，外层 List 代表所有顶点，内层 List 代表好友圈
        List<List<Integer>> graph = new ArrayList<>();

        // 我们有 4 个顶点，预先给每个人开辟一个空的好友圈
        int numVertices = 4;
        for (int i = 0; i < numVertices; i++) {
            graph.add(new ArrayList<>());
        }

        // 添加边: 0 连着 1 和 2
        graph.get(0).add(1);
        graph.get(0).add(2);

        graph.get(1).add(0); // 无向图双向添加
        graph.get(2).add(0);

        // 添加边: 1 连着 3
        graph.get(1).add(3);
        graph.get(3).add(1);

        // 查看顶点 1 的所有邻居
        System.out.println("顶点 1 的邻居有: " + graph.get(1)); // 输出 [0, 3]
    }
}
```

{% tab title="C++" %}
```cpp
#include <iostream>
#include <vector>

int main() {
    // C++ 面试中最经典的写法：vector 里 嵌套 vector
    int numVertices = 4;
    std::vector<std::vector<int>> graph(numVertices);

    // 添加边：0 和 1,2 有边； 1 和 3 有边
    // 注意：如果是无向图，你要从两面分别 push_back
    graph[0].push_back(1);
    graph[1].push_back(0);

    graph[0].push_back(2);
    graph[2].push_back(0);

    graph[1].push_back(3);
    graph[3].push_back(1);

    // 访问 1 的所有邻居
    std::cout << "顶点 1 的邻居有: ";
    for (int neighbor : graph[1]) {
        std::cout << neighbor << " "; // 输出 0 3
    }
    std::cout << std::endl;

    return 0;
}
```

**邻接表的优缺点：**

* **大优点**：极度省内存！微信有多少好友就记录多少次，对于没有任何交集的陌生人，连 1 byte 的内存都不会浪费。这也是工业界和各类算法竞赛首选的底层数据结构。
* **缺点**：想知道顶点 A 和能 B 能不能连通？你没办法像矩阵那样 O(1) 秒回了。你只能把 A 的整个好友列表从头到尾扫一遍（找里面有没有 B），查询某条具体边的速度略微变慢。但这在巨幅的内存节省面前，是非常值得的让步。

## 本章小结

* **图的核心认知**：面对错综复杂的“多对多”网状结构（现实世界绝大部分关系），必须改用由**顶点**和**边**拼成的“图结构”。
* **单行道与双向车道**：根据边是否具有单向箭头，图分为**有向图（例如微博关注）和无向图（例如微信互加好友）**。
* **代价各异**：如果在边上标着距离或收费额度，就变成了**带权图**，这是以后学习导航路线算法的基础。
* **如何存进电脑里（图的存储）**：
  * **邻接矩阵**：用巨大的二维数组暴力查阅。查询秒出，但浪费海量内存空间。
  * **邻接表**：就像手机通讯录一样，每个人只记录自己实实在在认识的人。极度省内存，也是 99% 的实战场合使用的最佳手段。
