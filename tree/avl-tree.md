# 平衡二叉树

在上一章中，我们见识到了二叉搜索树（BST）的强大：它能像“猜数字”游戏一样，以极快的速度（O(log n)）找到我们需要的数据。

但是，在上一章的结尾我们也点出了 BST 的一个**致命缺陷**：如果你运气极差，插入的数据是有序的（比如依次插入 1, 2, 3, 4, 5），那这棵树就会长歪，退化成一条单边链表。这时候它的查找速度就会一撸到底，退化成和普通遍历一样的 O(n)，这就完全失去了用树的意义。

为了解决这个“长歪”的问题，两位俄罗斯科学家 G. M. Adelson-Velsky 和 E. M. Landis 在 1962 年发明了计算机科学史上第一棵**自平衡二叉搜索树**，并用他们名字的首字母命名为：**AVL 树**。

## 什么是“平衡”？

如果用生活中的例子来解释，AVL 树就像是一个极其强迫症的**天平管理者**。

它给每个节点都增加了一个评估指标，叫作**平衡因子（Balance Factor）**。

> **平衡因子 = 左子树的高度 - 右子树的高度**

AVL 树定下的极其严格的规矩是：**任何一个节点的平衡因子，只能是 -1、0、1 这三个数字中的一个。** 也就是说，无论是哪个节点，它的左边和右边的子树高度差，**绝对不能超过 1 层**！一旦超过 1（比如左边比右边高了 2 层），这棵树就认为自己“失衡”了，它会立刻发动一种名&#x53EB;**“旋转（Rotation）”**&#x7684;绝技，哪怕把树干扭断重组，也要让自己重新恢复平衡。

## 神奇的魔法：旋转（Rotation）

当一棵树发现自己长歪了，它该怎么“掰直”自己呢？科学家发明了两种基本动作：**右旋（Right Rotation）** 和 **左旋（Left Rotation）**。

这是整个自平衡树最烧脑、但也最精妙的设计。对于入门者来说，你不要去死记硬背代码，而是要**把节点想象成用绳子连起来的猴子**。你需要提着某个猴子往下拉，同时把另一只猴子顺势提上来。

### 1. 右旋 (Right Rotation) —— 应对“左边太重”

假设我们依次插入了 `3, 2, 1`，树变成了向左倾斜的单边树：

```
      3 (失衡点，左树高2，右树高0。平衡因子为 2，严重失衡！)
     /
    2
   /
  1
```

此时“天平”严重向左倾斜。为了恢复平衡，我们需要对 `3` 进行**右旋**。 **动作要领**：把 `2`（左小弟）提拔上来当新老板，把 `3`（原来的老总）往右下方拽，变成 `2` 的右小弟。

```
       2 (新老板)
     /   \
    1     3 (被拽下来的老总)
```

看！树奇迹般地恢复了完美平衡！这就叫做一次“右旋”。

### 2. 左旋 (Left Rotation) —— 应对“右边太重”

假设我们依次插入了 `1, 2, 3`，树变成了向右倾斜的单边树：

```
  1 (失衡点，平衡因子 -2)
   \
    2
     \
      3
```

此时我们需要对 `1` 进行**左旋**。 **动作要领**：把 `2`（右小弟）提拔上来当新老板，把 `1`（原来的老总）往左下方拽，变成 `2` 的左小弟。

```
       2
     /   \
    1     3
```

同样，树再次恢复了对称的平衡。

### 3. 先左后右 / 先右后左 —— 应对“长得像闪电”的情况

有时候单靠一次旋转无法解决问题，比如下面这种“长得像闪电/拐弯（< 或 > 形状）”的树：

```
    3
   /
  1
   \
    2
```

如果你直接把 `3` 往右拽，`2` 该放哪呢？结构会乱套。 这种情况下，必须**打两套组合拳**：

1. **先左旋**：先对下面那个拐弯的节点（`1`）做一次左旋，硬生生把这道“闪电”拉直成一条左斜线（变成 `3 -> 2 -> 1`）。
2. **后右旋**：然后再对最上面那个失衡点（`3`）做一次干脆的右旋，让 `2` 上去当老板。

## AVL 树的旋转代码长什么样？（仅供感受）

由于手写 AVL 树的完整旋转代码极其枯燥，且充满各种指针断开重连的陷阱，通常只有专门应对高级算法面试或者写底层数据结构框架的人才会去手敲。

对于零基础学习者，你**只需要明白它的核心“篡位”机制即可**。

这里我们以 Python 为例，截取一小段“右旋”的核心逻辑让你感受一下它的画风：

```python
def right_rotate(y):
    # y 对应上面的 3（即将被拽下来的老总）
    # x 对应上面的 2（即将上位的新老板）
    x = y.left
    T2 = x.right # x 原本可能有自己的右小弟，必须先记下来，以免弄丢

    # 开始篡位（旋转）
    x.right = y  # 新老板 x 提拔上来，把老总 y 变成自己的右侧下属
    y.left = T2  # 由于 y 失去了原来的左下属 x，它刚好接手 x 原来的右小弟 T2

    # 如果节点有记录高度的字段，这里还需要重新计算 x 和 y 的高度...

    return x # 返回新老板，取代原来老总的位置
```

这段代码看似简单，但如果在实际工程中加上四种可能的情况判断（LL、RR、LR、RL），以及每个节点高度的实时更新，完整代码会变得相当臃肿。

## 工程界的王者：红黑树 (Red-Black Tree)

既然我们学会了 AVL 树，难道 Java 里的 `TreeMap`，C++ 里的 `std::map`，各种数据库用的底层搜索树，背后的源码都是 AVL 树吗？

**并不是的。**

AVL 树实在**太较真了**。它不允许任何一点点的“长短脚”（高度差永远保证 <= 1）。这就导致了如果数据更新很频繁（比如经常插入和删除节点），AVL 树就会跟发了疯一样，一天到晚都在疯狂地进行“左旋右旋”，极其耗费 CPU 资源，反而拖慢了整体的运行速度。

为了解决这个“过度调整”的问题，聪明的科学家退了一步，发明了业界大名鼎鼎的**红黑树（Red-Black Tree）**。

红黑树给节点涂上“红”和“黑”两种颜色，它**不再追求那种绝对完美的高度差**，而是追求一&#x79CD;**“大致平衡”**（它保证一棵树的最长路径，绝对不会超过它最短路径的两倍）。因为条件放宽了，红黑树在插入和删除数据时，就不需要那么频繁地触发旋转了。

### 常见搜索树对比总结

| 树的种类        | 核心设计哲学           | 查询效率                   | 新增/删除效率       | 现实应用场景                                  |
| ----------- | ---------------- | ---------------------- | ------------- | --------------------------------------- |
| **普通的 BST** | 大小排好就行，长歪了算我倒霉   | 可能退化成 O(n)             | 可能退化成 O(n)    | 教学演示，刷基础算法题                             |
| **AVL 树**   | 严重强迫症，高度差绝对不超过 1 | 极快（强行维持完美平衡） O(\log n) | 较慢（经常需要旋转调整）  | 查询极多、插入删除很少的场景                          |
| **红黑树**     | 宽容的实用主义，大致平衡就行   | 略微比 AVL 慢，但人类感知不到      | 极快（因为减少了旋转次数） | Java TreeMap, C++ Map, Linux 内核调度等工业级基石 |

## 本章小结

* 普通的二叉搜索树（BST）有**退化成单边链表**的致命风险。
* **自平衡二叉树（如 AVL 树）通过检测平衡因子**，一旦发现左右高度差超过 1，就会强制触发**左旋**或**右旋**，把树“掰直”。
* 旋转的本质就是**断开指针后重新认“父子”节点**，它分为单旋（左/右）和双旋（先左后右/先右后左）。
* AVL 树因为对平衡的追求过于苛刻，修改性能较差；工业界在需要有序查找结构时，最爱用的其实是舍弃了绝对平衡、换取更少旋转次数的**红黑树（Red-Black Tree）**。
